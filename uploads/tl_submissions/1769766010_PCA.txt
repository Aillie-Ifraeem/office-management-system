1️⃣ Permission Creep Analyzer

(Security + IT + Real company pain)

What is “Permission Creep”? (Simple)

Imagine an employee:

Joins company → gets basic access

Moves to another team → gets extra access

Gets promoted → gets even more access

Changes role again → old access is never removed

Over time, the employee has:
❌ access they don’t need anymore
❌ access to sensitive systems
❌ permissions nobody remembers giving

This slow accumulation is called permission creep.

Companies lose data, get hacked, or fail audits because of this.

What a Permission Creep Analyzer DOES

It does not give permissions.
It analyzes them.

Think of it as a security auditor bot.

It answers questions like:

Who has too many permissions?

Which permissions are never used?

What happens if this account is hacked?

Which permissions should be removed first?

Real Example (Very Important)

Employee: Ali

Time	Role	Access Given
Jan	Intern	Read-only dashboard
Mar	Developer	Database write access
Jul	Team Lead	Admin panel
Dec	Moved to QA	❌ Old permissions still active

Now Ali:

Can change production data

Can access admin settings

But doesn’t need any of that

Your system detects this.

How Companies Think (Not Technical)

Companies think in risk.

Your tool answers:

“If Ali’s account is compromised, how bad is it?”

Core Features (Human Language)
1️⃣ Permission History

Tracks:

When a permission was added

Why it was added

For which role

(Most systems don’t track WHY)

2️⃣ Usage Tracking

Checks:

Has this permission been used in last 30 / 90 days?

If not:

“Why does this person still have it?”

3️⃣ Role vs Permission Mismatch

Compares:

Current role

Current permissions

If mismatch:
🚨 flag it

4️⃣ Risk Scoring (Very Important)

Each permission has a danger level.

Example:

View dashboard → low risk

Delete users → high risk

Access financial data → very high risk

Total risk score = sum of dangerous permissions

5️⃣ “What If” Simulation

Simulates:

“If this user is hacked, what systems are exposed?”

This is very impressive in interviews.

Why This Is Rare in Student Projects

Needs thinking, not just coding

Requires relationships between data

Needs security mindset

That’s why recruiters love it.

You are the security guard of a huge office building.

Many rooms

When someone joins:

They move to another department → they get more keys

They get promoted → they get even more keys

They change job → nobody takes old keys back

Now years later:

Some people have keys to rooms they never enter

Nobody remembers why they have those keys

This is dangerous.

Now: What is YOUR project?
Your project is NOT making keys.
Your project is NOT opening doors.

👉 Your project is the guy with a clipboard who CHECKS KEYS.

What your project ACTUALLY DOES (Plain English)

Your Permission Creep Analyzer:

✅ Looks at who has which keys
✅ Checks which keys they actually use
✅ Checks whether their job matches the keys
✅ Calculates how dangerous their key set is
✅ Tells the boss where the risks are

That’s it. That’s the whole idea.

Let’s go step by step with this analogy
1️⃣ “Who has which keys?”

Your tool makes a list:

Ali has keys to:

Office room

Server room

Finance vault

2️⃣ “Which keys are actually used?”

It checks:

Ali enters server room every day ✔

Ali never enters finance vault ❌

“This key exists but is never used.”

3️⃣ “Does Ali’s job make sense for these keys?”

Ali’s job = Web Designer

Question:

Does a web designer need the finance vault key?

Probably not.

🚩 Your tool flags it.

4️⃣ “How bad is this situation?”

Some keys are more dangerous than others.

Office room → low risk

Server room → medium risk

Finance vault → VERY high risk

Your tool adds this up and says:

“Ali’s key risk = HIGH”

5️⃣ “What if Ali loses his keys?”

Your tool imagines:

“If someone steals Ali’s keys, which rooms can they enter?”

If answer = finance vault, alarm bells ring 🔔

That’s it. That’s your project.
